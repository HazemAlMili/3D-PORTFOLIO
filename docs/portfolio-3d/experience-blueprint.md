# Experience Blueprint
## Cinematic 3D Full Stack Developer Portfolio
## Concept: The System Behind Every Screen

Status: Phase 0 Draft
Purpose: Single source of truth for story, scenes, content, and visual direction.
Scene Count: 8
Navigation Model: Scroll-driven cinematic journey
Core Grammar: Approach → Enter → Immerse → Exit → Transfer

---

# 1. Creative Lock

* **Approved Concept Name:** The System Behind Every Screen
* **Meaning of the Concept:** The portfolio is a scroll-driven digital narrative showing that a modern senior developer understands the entire system backing every interface, rather than just constructing a visual front-end facade.
* **What the Experience Is:** A deterministic, scroll-driven cinematic walkthrough showing interactive screens on physical devices, which act as portals to showcase technical competencies, deep architectural design, and real-world project outcomes.
* **What the Experience Is Not:** The experience must not become a game, a random futuristic room, a spaceship, a city, or a generic 3D demo. It does not contain free-roam controls or unrelated sci-fi assets.
* **Non-Negotiable Storytelling Rule:** The portfolio must prove that the developer understands the system behind every interface, not only the visual screen.
* **Non-Negotiable Software Engineering Positioning:** Every scene must be dedicated to highlighting tangible full-stack expertise (e.g., APIs, database layouts, performance budgets, accessibility, systems architecture, and deployment pipelines) rather than purely decorative visuals.

---

# 2. Storytelling Grammar

The portfolio follows a strict, repeatable universal scene rhythm:
```
Approach → Enter → Immerse → Exit → Transfer
```

Every major scene must be entered, lived, and exited. Below are the definitions for each step:

* **Approach:** The camera moves towards the screen of a specific device or target system environment.
* **Enter:** The camera passes through the glass or surface of the device's screen, using it as a transition portal.
* **Immerse:** The 3D and overlay content becomes the active, readable world. The camera holds or moves slightly as the user scrolls to read specific content.
* **Exit:** The camera pulls back, exiting the portal world to reveal the physical device environment again.
* **Transfer:** The camera pans, rotates, or translates to transition from the previous device context to the next target environment.

---

# 2.1 Motion Grammar Lock

This section defines how the camera and scenes behave during scroll.

## Universal Motion Rules
* **5.1 Core Rule:** Every scene must be driven by scroll progress, not by hidden time-based state.
* **5.2 Reversibility Rule:** Every motion state must be reversible when the user scrolls upward.
* **5.3 Camera Rule:** The camera must behave like a deliberate cinematic operator, not a random orbit camera.
* **5.4 Reading Rule:** The camera must hold or slow down during content comprehension moments.
* **5.5 No Repetition Rule:** No two adjacent true portal-entry scenes may use the same transition variant.
* **5.6 No Game Rule:** The visitor must not be required to drive, fly, aim, or freely navigate the 3D world.
* **5.7 No Motion Over Content Rule:** Motion must never be more important than comprehension.

## Approved Portal Transition Variants
```
Variant 1 — Direct zoom into screen
Variant 2 — Screen surface expands into content world
Variant 3 — Screen pixels transform into content grid
Variant 4 — Device UI opens like a system window
Variant 5 — Data tunnel transition
Variant 6 — Interface frame dissolves into full section
Variant 7 — Camera passes through glass/screen reflection
```
*Note: Variant 7 is reserved for later polish only. It must not be required for the Phase 0 lock.*

# 4. Scene-by-Scene Blueprint

## Scene 01 — Opening / Seal Activation

### Purpose
Create the first premium impression, captivate the user immediately, and initialize the system environment.

### Story Meaning
The system is authenticated, initialized, and unlocked for exploration.

### Device / Environment
A dark, technical, abstract space. A central digital seal/stamp that resolves to connect with the main display device.

### Entry Moment
The experience starts in darkness with subtle, floating data structures and code fragments emerging from the black.

### Inside-Screen / Active Content
This is not an inside-screen content scene yet. It is the opening title and system boot bookend.

### Visual Elements
Dark graphite workspace, floating code syntax fragments, technical data particles, a detailed abstract seal/stamp asset, screen ripple/distortion shaders, and the main developer name/logo reveal.

### Background System Elements
Faint grid lines, subtle tracking data lines, boot-like console signals, and minimal code snippets compiling in the background.

### Motion Notes
Floating code fragments assemble into the central digital seal; the seal strikes forward; a radial grid ripple/distortion shader effect spreads outward; the logo/name is revealed; the camera pulls back to frame the main display device.

### Content Requirements
Developer's name/logo, primary role title ("Full Stack Developer / Software Engineer"), and a brief positioning hook.

### Must Avoid
Human coder avatars, cartoon stamps, heavy screen-shake, long unskippable animations, cluttered code rain, or neon/over-bloomed visuals.

### Exit / Handoff
The camera completes its pullback to reveal the main display screen, which establishes the context for Scene 02.

### Lock Criteria
First impression feels highly premium; logo, name, and role are instantly legible; the boot duration is short/capped; screen has zero clutter; transition into Scene 02 is natural.

### Locked Motion Grammar
* **Approach:** The experience begins in darkness with subtle code/data fragments appearing in depth.
* **Enter:** No true screen portal entry. This is a custom opening bookend.
* **Immerse:** Data fragments assemble into an abstract digital seal/stamp. The seal strikes the screen or spatial surface, producing a controlled ripple/distortion reveal.
* **Exit:** The camera pulls back from the seal/logo reveal.
* **Transfer:** The pullback reveals the main display context and prepares Scene 02.
* **Motion Risk:** The impact can become too aggressive or too game-like if shake, particles, or bloom are overused.
* **Motion Must Avoid:** No long unskippable intro. No human character coding. No cartoon stamp. No excessive screen shake. No code-rain cliché.

## Scene 02 — Main Display / Hero Identity

### Purpose
Introduce the developer's core identity, value proposition, and primary actions clearly.

### Story Meaning
The visitor immediately understands who the developer is and what value they bring, with immediate options to explore or convert.

### Device / Environment
A high-detail physical main display monitor.

### Entry Moment
The camera approaches the main display and zooms directly into the screen surface, converting it into a viewport portal.

### Inside-Screen / Active Content
Hero text cards, value propositions, and interactive primary call-to-actions (CTAs).

### Visual Elements
The physical display frame, soft ambient screen glow, premium typography overlay, and clean interactive interface lines.

### Background System Elements
System boot indicators, faint streaming data lines, mini code compile logs, and system initialization metadata.

### Motion Notes
Direct camera zoom into the screen surface (Variant 1 portal transition); text content animates into view smoothly after entry; CTAs fade in; camera transitions laterally toward the ultrawide monitor upon scrolling out.

### Content Requirements
Developer name, primary role title, core value proposition, "View Projects" CTA, "Contact Me" CTA, and quick links to GitHub and LinkedIn.

### Must Avoid
Vague or hiding identity, lengthy biographies, abstract/non-obvious value statements, unreadable typography, or small/faded CTAs.

### Exit / Handoff
The camera exits the screen portal of the main display and pans/slides sideways to align with the ultrawide monitor.

### Lock Criteria
Identity and role are clear within 3 seconds of entry; CTAs are prominently positioned; typography contrast is high; copy aligns with professional software engineering.

### Locked Motion Grammar
* **Approach:** The camera stabilizes in front of the main display after the opening pullback.
* **Enter:** Variant 1 — Direct zoom into screen.
* **Immerse:** Hero identity content appears clearly inside the screen. The camera should hold enough for reading the name, role, value proposition, and CTAs.
* **Exit:** The camera pulls out of the main display without snapping or teleporting.
* **Transfer:** The camera moves laterally toward the ultrawide architecture display.
* **Motion Risk:** The first true portal can become confusing if the zoom is too fast or if text appears before the camera settles.
* **Motion Must Avoid:** No fast push-in. No unreadable hero text. No hidden CTA. No abstract intro after the user already passed the opening.

## Scene 03 — Ultrawide / Architecture Mindset

### Purpose
Prove the developer's system-level design and full-stack software architecture capabilities.

### Story Meaning
The developer is not just an interface builder; they design robust, decoupled, and scalable full-stack pipelines.

### Device / Environment
An ultrawide monitor display.

### Entry Moment
The camera approaches the ultrawide screen as the device user interface opens up like a cascading system window.

### Inside-Screen / Active Content
Interactive visualization of an application architecture, from frontend layers to cloud infrastructure.

### Visual Elements
Decoupled backend, API, frontend, database, and infrastructure layers; connected node paths; moving data stream pulses.

### Background System Elements
Abstract service block containers, API route endpoints, database cylinder motifs, and deployment container models.

### Motion Notes
The device UI windows cascade open (Variant 4 portal transition); architecture layers build sequentially; animated pulse lines travel down data paths; camera locks on horizontal scroll to let the user review the system diagram.

### Content Requirements
Must document the following layers: Frontend (e.g., React, Next.js), API Layer (e.g., REST, GraphQL, Gateway), Backend Services (e.g., Node.js, Go microservices), Database (e.g., PostgreSQL, Redis), and Deployment / Infrastructure (e.g., Docker, AWS, CI/CD).

### Must Avoid
Fake node graphs, arbitrary meaningless icons, illegible diagram labels, static flat images, or confusing overpopulated networks.

### Exit / Handoff
The architecture window collapses or resolves back into a single glowing system data line, which guides the camera down toward the laptop device.

### Lock Criteria
The stack layers are immediately identifiable; all text labels remain legible; data flow matches real system logic; motion is purposeful and guides the reader's eyes.

### Locked Motion Grammar
* **Approach:** The camera aligns with the ultrawide monitor from the Scene 02 lateral transfer.
* **Enter:** Variant 4 — Device UI opens like a system window.
* **Immerse:** Architecture layers open in sequence: Frontend, API Layer, Backend Services, Database, Deployment / Infrastructure. Data pulses travel through the layers while the camera remains composed.
* **Exit:** The architecture window collapses, fades, or resolves into a single transferable data line.
* **Transfer:** That data line guides the camera toward the laptop/projects scene.
* **Motion Risk:** The architecture can become unreadable if too many nodes animate at once.
* **Motion Must Avoid:** No fake complexity. No random icons. No unreadable node galaxy. No flat boring poster. No animation that competes with layer labels.

## Scene 04 — Laptop / Projects Layer

### Purpose
Display real-world projects, verify technical skills, and build recruiter/client trust.

### Story Meaning
The visitor peers into the engineer's working machine to inspect shipped, production-grade applications.

### Device / Environment
A sleek laptop workstation.

### Entry Moment
The camera moves down to the laptop; the keyboard backlights illuminate and the screen wakes as pixels transform into a case-study content grid.

### Inside-Screen / Active Content
Interactive project case-studies and technical write-ups.

### Visual Elements
Detailed physical laptop mesh, project overview panels, code editor/terminal interface overlays, technical tags, and navigation arrows.

### Background System Elements
Miniature system architecture diagrams for each project, deployment status indicators, and Git commit logs.

### Motion Notes
Laptop wakes from sleep; screen pixels rasterize into a content grid (Variant 3 portal transition); individual project cards slide or fade in; user can navigate through projects; camera remains steady.

### Content Requirements
Showcase exactly 2–4 projects. Each project must feature: title, one-line summary, specific problem solved, technical stack used, what was built, architecture/engineering note, and measurable result/outcome, alongside active links to live demo and GitHub.

### Must Avoid
Static screenshots with no text, decorative cards lacking engineering detail, excess projects (>4), unverified claims, lorem ipsum, or vague descriptions.

### Exit / Handoff
Camera pulls out from the laptop screen and pans down towards the tablet lying next to the workstation.

### Lock Criteria
Case studies are scannable; engineering details (problems, architectures, stacks) are clear; external links are fully keyboard-accessible.

### Locked Motion Grammar
* **Approach:** The camera descends or angles toward the laptop, with a subtle keyboard/screen wake effect.
* **Enter:** Variant 3 — Screen pixels transform into content grid.
* **Immerse:** Project panels emerge as structured case-study windows. The camera holds while project content is read and navigated.
* **Exit:** The project grid deconstructs back into screen pixels or collapses into the laptop display.
* **Transfer:** The camera exits the laptop screen and moves toward the tablet.
* **Motion Risk:** Project content can become decorative if transitions dominate the actual case-study proof.
* **Motion Must Avoid:** No project cards with only title/image. No excessive panel movement while reading. No hidden Live/GitHub links. No tiny text. No carousel-only dependency without accessible controls.

# 4.1 Transition Continuity Rules

1. Every exit must visually explain how the next scene is reached.
2. Every transfer must preserve orientation; the visitor should understand where they came from and where they are going.
3. True portal entries must not repeat the same visual trick in adjacent scenes.
4. Content-heavy scenes must include a camera hold during the Immerse phase.
5. Scene 07 must intentionally break the device-screen rhythm to reveal backend/full-stack depth.
6. Scene 08 must slow the experience down and prioritize conversion clarity.
7. Reverse scrolling must restore the previous scene state without broken transitions.

---

# 5. Global Content Principles

* **Short and Precise:** Visitors scroll quickly. Copy must be structured as scannable titles, bullet points, or short sentences. Avoid paragraphs of prose.
* **Engineering-Centric:** Position all descriptions around technical problems, system design, architectural constraints, and quantifiable metrics.
* **No Hype / Forbidden Wording:** Do not use marketing buzzwords or generic templates.
  * *Forbidden language examples:* "passionate developer," "amazing websites," "modern solutions," "creative digital experiences," "high quality work."
  * *Preferred language direction:* Use precise tech stack terms, measurable outcomes, system architectures, database designs, and performance budgets.
* **Mini Case-Studies:** Projects must tell a brief story of problem, solution, tech stack, architectural decision, and quantifiable outcome.
* **Proof Over Claims:** Detail the exact metrics, testing patterns, and optimization rules followed (e.g. bundle size caps, Lighthouse targets).

---

# 6. Global Visual Principles

* **Palette & Atmosphere:** Smoked glass materials, dark graphite bases (#121214, #18181B), deep matte metal frames, and subtle dark blue backgrounds.
* **Accents:** Soft cyan/blue data accents (#00F0FF, #00A3FF) for active lines, data paths, and interactive highlights.
* **Typography:** Crisp, high-contrast white sans-serif headings and structured monospace data labels. Google Fonts like Inter or Outfit are preferred for readability.
* **Visual Restraint:** Avoid neon overloads, purple cyberpunk schemes, toy-like bright colors, heavy bloom shaders, massive particle counts, or fictional sci-fi HUD elements. Everything must look like a clean, real engineering workspace.

---

# 7. Global Motion Principles

* **Scroll Integration:** Scroll progress maps deterministically to the camera path. Scrolling backwards must traverse the path in reverse with zero state errors.
* **Camera Determinism:** The camera's transform must be calculated as a pure function of scroll progress. Do not use accumulated scroll deltas or time-based continuous updates for the core camera track.
* **Readability Guards:** The camera must hold position during the "immerse" segment of each scene to allow text to be comfortably read.
* **Story-Driven Transitions:** Every move must have a physical narrative logic (e.g., following a network data cable to the next device). Motion must never be more important than comprehension.

---

# 8. Accessibility and Fallback Principles

* **Reduced Motion:** If a user has `prefers-reduced-motion` enabled, the 3D scroll canvas is bypassed/hidden and replaced by a semantic, clean, high-contrast stacked HTML document containing all portfolio content.
* **WebGL Fallback:** In the absence of WebGL support, a clean fallback page displays the full portfolio contents semantically with standard web styling.
* **Quick Navigation:** Keyboard-accessible skip links allow users to jump directly to "Projects" or "Contact" without scrolling through the 3D path.
* **Semantic HTML:** All text inside screens must exist as real, indexable, screen-reader-accessible DOM elements overlaid precisely on top of the WebGL canvas, rather than textured on 3D meshes.
* **Contrast and Interaction:** Focus states must be highly visible; font contrast must meet WCAG AA standards.

---

# 9. Performance Principles

* **Single Shared Canvas:** The application uses a single persistent `<Canvas>` container, minimizing memory usage and avoiding context loss.
* **Lazy Loading & Preloading:** Device GLTF models are split and lazy-loaded. Model N+1 starts preloading only when Model N is in its active immerse phase.
* **Mesh Optimization:** Use compressed Draco GLTF geometries and KTX2/Basis textures.
* **Lighting and Shadows:** Keep active lights to 3 or fewer. Bake static shadows where possible to avoid expensive real-time shadow calculations.
* **Device Tiering:** A fast GPU tier detection check (e.g., `detect-gpu`) classifies the client system. Lower-tier systems automatically disable dynamic shadows, reduce particle limits, and disable post-processing.

---

# 10. Open Decisions and Dependencies

* **Personal Details:** Final developer name, logo branding, and exact email.
* **Copywriting:** Final project details (problems, stacks, outcomes) and final value proposition text.
* **Static Assets:** The production of the actual device GLB files (main display, ultrawide, laptop, tablet, phone, server node) or equivalent procedural mesh generators.
* **CV File:** Real CV document placed at the CV download location.
* **Camera Keyframes:** Precise numeric `position` and `target` values for `CameraPose` keyframes (these can only be finalized once physical model scale and bounding boxes are determined).
